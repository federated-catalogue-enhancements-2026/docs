== Architecture Decisions

=== ADR 1: Monolithic vs. Microservices
**Context:** Are the subcomponents implemented as separate micro-services or are they integrated into a single service.

1. **Microservice**: Usage of different technologies possible, independent development, scalability, ...
2. **Single Service**: Easier maintainability, understanding, easier to handle concurrency (especially conflicting operations like adding an asset and changing schema)

**Decision:** Implement catalogue as a modular, single service. It's easier to integrate the different components. Most components are called by only one other component. A distributed microservice implementation only introduces additional communication overhead. Scalability issues are not expected, since the main load is read-only and time consuming operations are queries against the graph db, which is a separate component that can be horizontally scaled.
Therefore, the Federated Catalogue is considered as microservice application within the GXFS ecosystem. The federated catalogue itself also consists of different, individually deployable components.

=== ADR 2: Intra Catalogue communication
**Context:** If the catalogue is implemented as one component, the subcomponents can communication using Method-calls. Another option is to implement an internal event-/notification system, whereby each component registers itself as a listener on other components or a central message bus.

1. **Method-Calls:** Easy to implement, clear structure and easy to debug. Limited options for extendability.
2. **Event-based:** Highly plugable systems, where new components can be added without other components knowing that. Additional indirection which causes complexity. Harder to debug.

**Decision:** The Catalogue es a static, pre-specified and configured component. Adding plugins during runtime is not needed. The maintainability of the catalogue is more important than additional flexibility.

=== ADR 3: Disable Signature Verification by Default

**Context:** The verification pipeline has four independent flags: `semantics`, `schema`, `vp-signature`, and `vc-signature`. Signature verification resolves the `verificationMethod` DID, retrieves the public key, and verifies the JWS proof. However, without the Gaia-X trust framework (`gaiax.enabled=true`), the `x5u` certificate chain that binds the DID to a vetted legal entity is never validated.

Without the trust anchor chain, signature verification only proves internal consistency (the credential was signed by whoever controls that DID) and tamper detection (the payload has not been modified). It does not establish identity, trust, or accountability — anyone can create a `did:web` on a server they control and sign arbitrary claims.

This makes signature verification without trust anchor enforcement ineffective as a security measure. Meanwhile, keeping signatures enabled by default forces all deployments to stand up DID resolution infrastructure even when not using the Gaia-X trust framework.

**Decision:** Disable `vp-signature` and `vc-signature` in the default application profile (`application.yml`). Signature verification is re-enabled in the strict deployment profile alongside the full trust framework (Gaia-X enabled, schema validation, trust anchor registry).

[cols="1,1,1"]
|===
| Flag | Default profile | Strict profile

| `semantics` | `true` | `true`
| `schema` | `false` | `true`
| `vp-signature` | `false` | `true`
| `vc-signature` | `false` | `true`
| `gaiax.enabled` | `false` | `true`
|===

This aligns with CAT-FR-SF-04 (remove forced validation): validation flags that provide no security value without the trust framework should not be forced by default. Production deployments must explicitly enable signatures together with the trust framework.

=== ADR 4: Non-RDF Asset Storage Strategy

**Context:** The catalogue needs to support non-RDF assets (PDFs, plain JSON, YAML, images, binary files) alongside RDF credentials. Several approaches were considered:

1. **Separate `/assets` endpoint and storage:** New endpoint, new database table, new service layer. Clean separation but duplicates lifecycle management, filtering, and metadata handling.
2. **Unified endpoint with dual-path routing:** Extend the existing `POST /assets` endpoint (formerly `POST /self-descriptions`) to accept non-RDF content. Reuse the `assets` table (formerly `sdfiles`, extended with asset metadata columns), the `AssetStore` interface, and the existing filter/lifecycle infrastructure. Route to verification or direct storage based on content-type detection.
3. **External object storage (S3/MinIO):** Delegate non-RDF storage to an external system. Adds infrastructure complexity and a new dependency.

**Decision:** Option 2 — unified endpoint with dual-path routing.

* The `POST /assets` endpoint accepts `multipart/form-data` and `application/octet-stream` in addition to `application/json` and `application/ld+json`.
* An `RdfDetector` classifies uploads by content type (configurable allowlist) and content inspection (JSON-LD `@context` pattern).
* RDF content follows the existing verification + graph storage path.
* Non-RDF content bypasses verification and graph storage entirely. It is stored in a dedicated `assetFileStore` (separate directory from schema/context caches) with metadata in the `assets` table.
* Three columns are added to `assets`: `content_type` (VARCHAR), `file_size` (BIGINT), `original_filename` (VARCHAR). The `content` column is made nullable (non-RDF assets store content only in the File Store).
* Non-RDF assets receive auto-generated IRI identifiers: UUID URNs (`urn:uuid:...`) per RFC 4122, assigned by `IriGenerator` (see ADR 7).

**Rationale:** Reusing the existing endpoint and metadata infrastructure avoids duplication of lifecycle management, filtering, pagination, and access control. The `RdfDetector` allowlist gives operators explicit control over which content types trigger verification. Separate File Store instances keep asset blobs isolated from schema caches for operational clarity.

Non-RDF assets can be subsequently enriched with graph-queryable RDF metadata via the same `POST /assets` endpoint (see ADR 15).
To distinguish the original upload modality from the post-enrichment state, a `content_kind` column (`RDF` / `NON_RDF`) was added to the `assets` table. `content_kind` is set at first upload and never changed by enrichment.

=== ADR 5: Fine-Grained Permission-Based Authorization

**Context:** The original authorization model used four coarse-grained roles (Ro-MU-CA, Ro-MU-A, Ro-AS-Afootnote:[Originally named `Ro-SD-A` (Self-Description Administrator). Renamed to `Ro-AS-A` (Asset Administrator) to align with the generic asset terminology.], Ro-PA-A) to control API access. This presented several limitations:

1. **No read-level control.** Asset and schema read endpoints were open to any authenticated user, with no way to restrict read access.
2. **No separation of concerns.** The Ro-AS-A and Ro-MU-A roles granted full write access to assets, with no way to differentiate between create, update (revoke), and delete operations.
3. **No schema write control.** Schema management was restricted only to Ro-MU-CA, with no intermediate access level for schema administrators.
4. **No query access control.** The query endpoint was accessible to any authenticated user.

Two approaches were considered:

1. **Extend existing roles:** Add more coarse-grained roles (e.g., Ro-Schema-A). Simple but doesn't address the granularity problem and leads to role proliferation.
2. **Fine-grained permissions with composite roles:** Define atomic permissions (ASSET_CREATE, ASSET_READ, etc.) and compose them into roles. Maximum flexibility while maintaining backward compatibility.

**Decision:** Option 2 — fine-grained permission-based authorization.

Ten atomic permissions are defined: `ASSET_CREATE`, `ASSET_READ`, `ASSET_UPDATE`, `ASSET_DELETE`, `SCHEMA_CREATE`, `SCHEMA_READ`, `SCHEMA_UPDATE`, `SCHEMA_DELETE`, `QUERY_EXECUTE`, and `ADMIN_ALL`. Each API endpoint requires a specific permission (or `ADMIN_ALL`). Permissions are implemented as Keycloak client roles and flow into Spring Security authorities via the existing `CustomJwtAuthenticationConverter`.

Backward compatibility is maintained by converting the legacy roles (Ro-MU-CA, Ro-MU-A, Ro-AS-A) into Keycloak composite roles that include the appropriate permissions. Users with legacy roles automatically receive the corresponding fine-grained permissions without any migration.

**Rationale:** Fine-grained permissions enable precise access control for multi-tenant deployments where different user groups need different levels of access to catalogue resources. The composite role mechanism ensures zero-disruption migration for existing deployments. No changes to JWT extraction or authentication infrastructure were required.

=== ADR 6: Claim Class Naming Convention

**Context:** CAT-NFR-01footnote:cat-enhancement[Enhancement of the XFSC Federated Catalogue. 2025. https://github.com/eclipse-xfsc/docs/blob/9a178c30f76610e5924fd72c39d259d6a8ae0461/federated-catalogue/src/docs/CAT%20Enhancement/CAT_Enhancement_Specifications%20v1.0.pdf] renamed the legacy `SdClaim` class (tied to Self-Description terminology). The team needed to decide on the new name. Three options were considered:

1. **`Claim`** — generic, forward-looking, but too broad for the current implementation which only extracts claims from credentials.
2. **`AssetClaim`** — aligns with the generic asset terminology but is misleading because non-RDF assets cannot have claims.
3. **`CredentialClaim`** — accurate for the current scope (only credentials produce claims), with a planned rename when scope expands.

The discussion established the following asset taxonomy:

[mermaid]
....
classDiagram
    class Asset
    class NonRDFAsset
    class RDFAsset
    class Credential
    class NonCredential
    class Claim {
        <<concept>>
    }

    Asset <|-- RDFAsset
    Asset <|-- NonRDFAsset
    RDFAsset <|-- Credential
    RDFAsset <|-- NonCredential
    Credential ..> Claim : contains
    NonCredential ..> Claim : contains

    note for Claim "A Claim is the atomic unit\nof semantic assertion (Triple)."
....

Key concepts:

* Non-RDF assets cannot have claims — they are opaque binary content.
* RDF assets can be credentials (verified VPs/VCs) or non-credentials (plain Turtle, RDF/XML).
* All RDF assets contain triples, and claims are synonymous with triples in this context.
* Even non-credential RDF assets contain claims; they are simply unverified.

**Decision:** Option 3 — `CredentialClaim`, with a phased naming strategy:

[cols="1,2,2",options="header"]
|===
| Phase | Class Name | Rationale
| Legacy | `SdClaim` | Original name tied to Self-Description terminology.
| Current | `CredentialClaim` | Credentials claims use this marker class to differentiate them from generic triples.
| Current | `RdfClaim` | Claims (Triples) that originate from any (non-credential) RDF asset. Superclass of `CredentialClaim`.
|===

**Rationale:** Naming for the current scope avoids premature abstraction.
The rename to `RdfClaim` was doen while adding non-credential claims extraction.

=== ADR 7: IRI-Based Asset Identification

**Context:** CAT-FR-AM-02: Each Asset MUST be identified by an IRI. DIDs and UUIDs as URNs (RFC 4122) MUST be supported.footnote:cat-enhancement[] The previous implementation used content-hash-based identifiers (`urn:asset:sha256:<hash>`) for non-RDF assets and `credentialSubject` for RDF credentials. Three approaches were considered:

1. **Keep hash-based identifiers:** Simple but violates CAT-FR-AM-02 — content hashes are not IRIs in the W3C sense, and they couple identity to content (any content change produces a new identifier).
2. **UUID URNs for all assets:** Assign a UUID URN (`urn:uuid:...`) per RFC 4122footnote:[https://www.rfc-editor.org/rfc/rfc4122[RFC 4122 — A Universally Unique IDentifier (UUID) URN Namespace]] to every asset regardless of type. Simple and uniform but discards the meaningful identifiers already present in RDF content (DIDs, ontology IRIs).
3. **IRI extraction with UUID fallback:** Extract IRIs from RDF content where possible (`credentialSubject.id`, `@id`, ontology/shapes/vocabulary IRIs), generate UUID URNs for non-RDF assets and as fallback for RDF content without an extractable IRI.

**Decision:** Option 3 — IRI extraction with UUID URN fallback.

Two new components are introduced:

* **`IriGenerator`** — extracts IRIs from RDF content (Verifiable Credentials via `credentialSubject.id`, generic JSON-LD via `@id`, OWL ontologies, SHACL shapes, SKOS concept schemes) or generates UUID URNs per RFC 4122 for non-RDF assets and as fallback.
* **`IriValidator`** — checks IRI formats via `isValid(iri): boolean`. Supported formats: DIDs (`did:<method>:<id>`), UUID URNs (`urn:uuid:<uuid>`), generic URNs (`urn:<nid>:<nss>`), and HTTP/HTTPS IRIs. Returns true/false — callers decide how to handle invalid IRIs.

The public API is extended with IRI-based methods (`getById`, `getFileById`) alongside the existing hash-based methods. Read endpoints use `/assets/{id}` where `{id}` is the URL-encoded IRI. Mutation operations (delete, revoke) remain hash-based — see the Hash-Based Mutation Operations Exception below.

The `federated-catalogue.asset.subject-id-prefix` configuration property is removed — IRI generation is now handled entirely by `IriGenerator`.

**Rationale:** Extracting meaningful IRIs from RDF content preserves semantic identity (a DID or ontology IRI carries provenance information that a UUID does not). UUID URNs for non-RDF assets satisfy RFC 4122 and CAT-FR-AM-02 while providing unique, stable identifiers without coupling identity to content. The content hash is retained internally for deduplication and file storage but is no longer the public identifier.

==== Hash-Based Mutation Operations Exception

The delete and revoke operations are the deliberate exceptions to IRI-based identification — both retain hash-based targeting:

* `DELETE /assets/{asset_hash}` — permanently removes an asset
* `POST /assets/{asset_hash}/revoke` — changes lifecycle status to REVOKED

The same rationale applies to both operations:

1. **Data model constraint** The `subjectid` column (the IRI) is not globally unique. The partial unique index `idx_asset_active` enforces uniqueness only among rows with `status=0` (ACTIVE). Multiple rows can share the same `subjectid` when they have different statuses (DEPRECATED, REVOKED, EOL). Mutation via `WHERE subjectid = :id` could match multiple rows — this is unacceptable for destructive or state-changing operations.

2. **Mutation semantics** Delete and revoke are irreversible operations that demand unambiguous single-row targeting. The `asset_hash` column is the PRIMARY KEY — it guarantees exactly one row is affected. The IRI identifies a _logical asset_; the hash identifies a _specific content artifact_. Mutations operate at the physical artifact level, not the logical level. For revoke specifically, hash-based lookup finds the asset regardless of its current status, enabling the business logic to return a proper 409 Conflict (e.g., attempting to revoke an already deprecated asset) rather than a misleading 404.

3. **Versioning readiness (CAT-FR-LM-01)footnote:cat-enhancement[]** The planned version control feature will introduce multiple version rows per `subjectid`, each with a different `asset_hash` and status. IRI-based mutation would become increasingly ambiguous — which version should be affected? Hash-based targeting remains precise regardless of how many versions exist. When versioning is introduced, both endpoints will be replaced by version-specific operations (`POST /assets/{id}/versions/{v}/revoke`).

4. **Audit versioning forward-look** When audit history tracking is introduced (e.g., Hibernate Envers), physical deletion conflicts with audit trail immutability. At that point, the delete operation should transition to a lifecycle status change (e.g., → EOL) rather than physical row removal. Retaining hash-based targeting keeps the migration path clean: `DELETE WHERE hash` transitions to `UPDATE status WHERE hash`. The targeting mechanism (hash) stays consistent across this evolution.

=== ADR 8: W3C VC-JOSE-COSE as Canonical JWT Header Standard

**Context:** The Federated Catalogue needs to accept JWT-wrapped credentials from Gaia-X Loire deployments. Two competing header value sets exist for JWT `typ` and `cty` fields:

1. **W3C VC-JOSE-COSEfootnote:[https://www.w3.org/TR/vc-jose-cose/[Securing Verifiable Credentials using JOSE and COSE. W3C Recommendation.]]:** `vc+jwt`, `vp+jwt` (typ) and `vc`, `vp` (cty). All values are IANA-registered.
2. **Gaia-X ICAM 24.07footnote:[https://docs.gaia-x.eu/technical-committee/identity-credential-access-management/24.07/credential_format/[Gaia-X ICAM 24.07 — Credential Format]]:** `vc+ld+json+jwt`, `vp+ld+jwt` (typ) and `vc+ld+json`, `vp+ld+json` (cty). None are IANA-registered.

The live GXDCH Compliance Service (`compliance.lab.gaia-x.eu/v2`) issues JWTs with W3C VC-JOSE-COSE headers, not the ICAM 24.07 headers. Accepting only ICAM headers means real Gaia-X credentials from GXDCH are rejected.

Three approaches were considered:

1. **Accept only ICAM 24.07 values:** Matches the Gaia-X spec letter but rejects live GXDCH credentials.
2. **Accept only W3C VC-JOSE-COSE values:** Correct per IANA registration but breaks any issuer that follows ICAM 24.07 literally.
3. **Accept both, canonicalize W3C:** Accept both header sets in code. Advertise only W3C values in public documentation (OpenAPI spec, error messages, architecture docs). Tolerate ICAM values silently.

**Decision:** Option 3 — W3C VC-JOSE-COSE as canonical, ICAM 24.07 tolerated.

[cols="2,2,2",options="header"]
|===
| Header | W3C VC-JOSE-COSE (canonical) | ICAM 24.07 (tolerated)
| VC `typ` | `vc+jwt` | `vc+ld+json+jwt`
| VC `cty` | `vc` | `vc+ld+json`
| VP `typ` | `vp+jwt` | `vp+ld+jwt`
| VP `cty` | `vp` | `vp+ld+json`
|===

HTTP content types follow the same principle:

[cols="1,2",options="header"]
|===
| Standard | Content-Type
| W3C (canonical) | `application/vc+jwt`, `application/vp+jwt`
| ICAM 24.07 (tolerated) | `application/vc+ld+json+jwt`, `application/vp+ld+json+jwt`
|===

**Rationale:**

* The W3C values are IANA-registered and used by the live GXDCH Compliance Service — the primary credential issuer in the Gaia-X ecosystem.
* The ICAM 24.07 values are not IANA-registered (`+ld` is not a valid structured syntax suffix per RFC 6838) and no known credential issuer in production uses them.
* Accepting both in code costs nothing (a `Set.of(...)` with additional values) while maximizing interoperability.
* Public-facing documentation advertises only the standards-compliant values, steering the ecosystem toward IANA-registered types.
* No cross-family validation is enforced: `typ` and `cty` are validated independently against their respective sets, not as pairs. A JWT with `typ: vc+jwt` and `cty: vc+ld+json` is accepted — this is intentionally permissive.
* `cty` remains mandatory despite the W3C spec stating SHOULD (not MUST). The live GXDCH always sends it, and relaxing the requirement would widen the surface for malformed JWTs.

=== ADR 9: Loire 2511 Ontology and Removal of Tagus Support

**Context:** The Gaia-X ecosystem has evolved through several trust framework generations. The earlier releases — Elbe and Tagus (ICAM 22.10) — used the W3C VC 1.1 data model with Linked Data proofs (`JsonWebSignature2020`) and the `gax-core:` / `gax-trust-framework:` namespaces. The current Loire release (ICAM 24.07) uses the W3C VC 2.0 data model with JWT signatures and the `\https://w3id.org/gaia-x/2511#` namespace.

The Federated Catalogue initially supported both generations in parallel (namespace coexistence via per-format type URI resolution). However, the Tagus-era class hierarchies are **disconnected graphs** from the 2511 hierarchy — maintaining both added configuration complexity (`legacy-type` properties) and code paths (`GAIAX_V1_TAGUS` enum, `Vc11Processor`, `UniSignatureVerifier`) with no active consumers.

**Decision:** Remove Tagus and Elbe support entirely. The catalogue exclusively supports the Loire 2511 namespace and VC 2.0.

[IMPORTANT]
====
**Migration impact:** This is not a graceful deprecation. The `FormatDetector` rejects any credential that uses the VC 1.1 context URI (`\https://www.w3.org/2018/credentials/v1`) or lacks a VC 2.0 context URI. Linked Data proofs are rejected with `"Linked Data proof verification is not supported. Use JWT or Enveloped Credential format."` Existing Tagus-era credentials cannot be uploaded to the catalogue — they must be re-issued in Loire/VC 2.0 format.
====

The ontology bundle contains only Loire 2511 files:

[cols="1,2",options="header"]
|===
| Type | File
| OWL ontology | `gx-2511.ttl`
| SHACL shapes | `gx-2511-shapes.ttl`
|===

Type URI configuration uses a single namespace:

[source,yaml]
----
participant:
  type: "https://w3id.org/gaia-x/2511#Participant"
resource:
  type: "https://w3id.org/gaia-x/2511#Resource"
service-offering:
  type: "https://w3id.org/gaia-x/2511#ServiceOffering"
----

Format-aware resolution in `CredentialVerificationStrategy.resolveBaseClass()`:

* `GAIAX_V2_LOIRE` → uses `type` (2511 URIs)
* `VC2_DANUBETECH` → skips Gaia-X type resolution entirely (domain-specific ontologies)

==== Stripped Ontology: Why the Registry OWL Cannot Be Used Directly

The official Gaia-X 2511 OWL file from the registryfootnote:[https://registry.lab.gaia-x.eu/development/owl/2511[Gaia-X Registry — OWL 2511]] is ~2.9 MB (~58,000 lines) and cannot be bundled as-is. It contains approximately 7,000 enum individuals whose IRIs embed a second `#` in the fragment component (e.g., `\https://w3id.org/gaia-x/2511#CountryNameAlpha2#SM`). These are not valid IRIs per RFC 3987footnote:[https://www.rfc-editor.org/rfc/rfc3987[RFC 3987 — Internationalized Resource Identifiers (IRIs)]]. Apache Jena emits one WARN log line per invalid IRI, which causes two problems:

1. **Test suite noise:** Thousands of warnings bury real errors in log output, making test failures difficult to diagnose.
2. **Apparent hangs:** The volume of log output slows test execution to the point where the suite appears to hang.

The catalogue's `SchemaStoreImpl` only needs the **class hierarchy** (`owl:Class` declarations and `rdfs:subClassOf` relationships) for type resolution via `ClaimValidator.getSubjectType()`. Data properties, annotation properties, OWL restrictions, and enum individuals are not queried by the type resolution SPARQL and are therefore unnecessary.

The bundled `gx-2511.ttl` is a **stripped extraction** containing only:

* The `owl:Ontology` declaration — required because `SchemaStoreImpl.analyzeAndValidate()` classifies files by checking for `owl:Ontology`, `sh:NodeShape`, or `skos:ConceptScheme`; without it the file would not be recognized as an ontology
* All `owl:Class` declarations
* All direct `rdfs:subClassOf` relationships — the extraction script handles the 2511 source pattern where blank-node OWL restrictions are interleaved with direct superclass references (e.g., `rdfs:subClassOf [ a owl:Restriction ; ... ], gx:ComputeServiceOffering`)

The following constructs are intentionally omitted:

[cols="2,3",options="header"]
|===
| Omitted construct | Effect
| Enum individuals (~7,000) | Carry the invalid double-`#` IRIs. Not used by type resolution.
| `owl:Restriction` blank nodes | Property cardinality/value constraints on instances. Not queried by `getSubjectType()`.
| `owl:equivalentClass` | Would cause the OWL reasoner to infer mutual `rdfs:subClassOf`. Currently absent from the 2511 core hierarchy.
| `owl:intersectionOf` / `owl:unionOf` | Inferred subclass from set-theoretic definitions. Not present in 2511.
| Data/annotation properties | Not queried by type resolution.
|===

The extraction is performed by `fc-tools/extract-ontology-hierarchy.py`, which parses the full registry TTL and writes a minimal hierarchy file. When the Gaia-X registry publishes updated 2511 ontology versions, the extraction must be re-run and tests verified:

[source,bash]
----
# 1. Download the new full OWL file
curl -o /tmp/gx-full.ttl https://registry.lab.gaia-x.eu/development/owl/2511

# 2. Run the extraction script (from the federated-catalogue repo root)
python3 fc-tools/extract-ontology-hierarchy.py \
    --input  /tmp/gx-full.ttl \
    --output fc-service-core/src/main/resources/defaultschema/ontology/gx-2511.ttl \
    --ontology-iri https://w3id.org/gaia-x/2511 \
    --source-url https://registry.lab.gaia-x.eu/development/owl/2511

# 3. Run tests to verify the hierarchy is intact
mvn test -pl fc-service-core -Dtest=LoireTypeResolutionTest,SchemaValidationServiceTest,SchemaStoreTest
----

[NOTE]
====
If a future 2511 spec version introduces `owl:equivalentClass`, `owl:intersectionOf`, or `owl:unionOf` for the core classes (`Participant`, `ServiceOffering`, `Resource`), type resolution will silently break because the stripped ontology omits these constructs. The extraction script must be updated to preserve them in that case.
====

[IMPORTANT]
====
**Production upgrade caveat:** `SchemaStoreImpl.initializeDefaultSchemas()` only loads bundled ontology and SHACL files when the schema table is empty (`dao.getSchemaCount() == 0`). On existing deployments that already have schemas in PostgreSQL, adding new 2511 files to the classpath will **not** trigger auto-loading on restart. The new ontology and shapes are silently ignored, and Loire type resolution fails without error. To pick up the new files, either clear the schema table before restart (via the Schema Store API or a Liquibase migration) or start with a fresh database.
====

**Rationale:**

* Removing Tagus support eliminates the disconnected-graph problem entirely — no `legacy-type` configuration, no per-format URI branching for Tagus.
* The 2511 OWL and SHACL files are bundled as classpath resources (fetched from the Gaia-X registry at development time), avoiding a runtime dependency on registry availability. The OWL file is stripped to class hierarchy only to avoid Jena IRI parse warnings from invalid enum individual IRIs (see above).
* The JSON-LD context for 2511 is configured via `doc-loader.additional-context` (fetched and Caffeine-cached at startup).
* Non-Gaia-X VC 2.0 (`VC2_DANUBETECH`) intentionally bypasses Gaia-X type resolution — future trust framework support for non-Gaia-X federations will add pluggable type resolution.
* The SRS preamble envisions the catalogue as a generic, trust-framework-agnostic metadata store. Removing a single legacy trust framework generation aligns with this direction — the catalogue's verification pipeline is designed to be pluggable, not permanently coupled to any specific Gaia-X release.

=== ADR 10: Trust Framework Bundles and Dynamic Role Resolution

**Related:** ADR 8, ADR 9

**Context:** ADR 9 introduced per-format type URI resolution to handle the disconnected Tagus/Loire class hierarchies.
The type URIs (`gx:Participant`, `gx:ServiceOffering`, `gx:Resource`) were carried in `application.yml` properties and mapped to a compile-time `TrustFrameworkBaseClass` enum.
This design has two operational failure modes and one structural limit:

1. **Sibling classes in the same ontology.** The Loire 2511 ontology places `gx:DigitalServiceOffering` as a direct child of `gx:GaiaXEntity` — not under `gx:ServiceOffering`.
SPARQL `rdfs:subClassOf*` traversal from `gx:ServiceOffering` never reaches it.
Credentials of type `gx:DigitalServiceOffering` (which has its own `sh:targetClass` SHACL shape and is a valid first-class credential type) were rejected with "no proper CredentialSubject found." The same problem recurs for any future sibling class.

2. **Cannot support multiple trust frameworks.** The catalogue is intended to be a generic federated catalogue (Gaia-X, future Catena-X, others).
Hardcoded Gaia-X URIs and a fixed Java enum cannot accommodate this without code changes per framework.

3. **Wrong abstraction level.** `TrustFrameworkBaseClass.SERVICE_OFFERING` was a Gaia-X concept baked into the type system, API routing, and storage.
It cannot meaningfully represent the equivalent concept in another trust framework, nor future Gaia-X classes that are siblings rather than subclasses of `gx:ServiceOffering`.

Three options were considered:

1. **Add more hardcoded URIs:** Add `gx:DigitalServiceOffering` as a second `service-offering.type` URI.
Fixes DSO immediately but deepens every structural problem above.

2. **SHACL-driven discovery, retain enum:** Replace hardcoded URIs with runtime discovery; keep `TrustFrameworkBaseClass` enum as the routing label.
Fixes DSO and enables new frameworks via config, but the fixed enum still couples the API and storage schema to Gaia-X concepts.

3. **Bundle-per-framework with explicit role declarations (chosen):** Each trust framework is a self-contained bundle (ontology + SHACL + a YAML config that declares the framework's roles), packaged with the JAR or supplied at deployment time as a filesystem directory.
At startup the catalogue loads all bundles, builds a `<typeUri> → <ResolvedRole>` index by walking each declared role's subclass closure, and routes credentials by role string.
No trust-framework concept exists in Java source — only in the bundle configs.

**Decision:** Option 3 — bundle-per-framework with explicit role declarations and dynamic subclass-closure indexing.

**Bundle layout:**

----
trustframeworks/
  <bundle-id>/
    framework.yaml      # bundle metadata and role declarations
    ontology.ttl        # OWL class hierarchy (optional for non-SHACL bundles)
    shapes.ttl          # SHACL shapes (optional for non-SHACL bundles)
----

Bundles ship from two sources:

* **Classpath** (`classpath:trustframeworks/*/framework.yaml`): bundles packaged in the JAR; scanned first.
* **Filesystem overlay directory** (optional, configured via `FEDERATED_CATALOGUE_TRUST_FRAMEWORKS_OVERRIDE_PATH`): a runtime-configurable directory whose subdirectories follow the same `<bundle-id>/framework.yaml` layout.
When set, the loader scans it after the classpath and merges its entries into the classpath result.

A bundle that fails to load is logged at WARN and skipped — it does not abort loading of remaining bundles.

**`framework.yaml` format:**

[source,yaml]
----
id: gaia-x-2511                          # framework profile id (registry key)
family: gaia-x                           # framework family (groups related profiles)
namespace: "https://w3id.org/gaia-x/2511#"
validation_type: shacl                   # shacl | json-schema | unknown

roles:
  Participant:                           # role name = OWL local name; URI = namespace + roleName
    additional_roots: [ ]                # extra URIs mapped to this role (non-OWL-subclass siblings)
    types: [ ]                           # forward-compat for json-schema engines (unused for shacl)
  ServiceOffering:
    additional_roots:
      - "https://w3id.org/gaia-x/2511#DigitalServiceOffering"
    types: [ ]
  Resource:
    additional_roots: [ ]
    types: [ ]

properties: { }                          # framework-specific metadata (reserved)
----

**Deployment-time overlays:**

The filesystem overlay directory exists to customise bundle properties — most commonly the URL of an external service the bundle calls, such as `properties.trust_anchor_url` — without rebuilding the JAR.
It also allows new bundles to be added at deployment time for environments that need them.

Resolution rule for filesystem entries:

* The filesystem `framework.yaml` is parsed for its `id` field first.
If `id` is missing or blank the entry is skipped with a warning.
* If `id` matches a classpath bundle, the filesystem entry is treated as an **overlay**: it is deep-merged onto the classpath base.
The filesystem file therefore only needs to declare `id` plus the fields it changes.
* If `id` is new (no classpath match), the entry is treated as a **new bundle** and must declare all mandatory fields (`id`, `family`, `namespace`, `validation_type`, `roles`).

Merge semantics for overlays:

* Nested maps (e.g. `properties:`, `roles:`) merge recursively; the filesystem value wins at each leaf key.
* Lists (e.g. `additional_roots`, `types`) and scalars (e.g. `namespace`, `validation_type`) replace wholesale when the filesystem provides them.
* Sibling files (`ontology.ttl`, `shapes.ttl`) are inherited from the classpath bundle when the overlay directory does not ship its own.

Minimal overlay example — redirect the Gaia-X bundle's trust anchor URL to an in-cluster service without copying the rest of the bundle:

[source,yaml]
----
id: gaia-x-2511
properties:
  trust_anchor_url: "https://did-server/api/trustAnchor/chain/file"
----

**Role declaration is a manual, per-framework curation step — not auto-derivable.**
The `roles:` map is the bundle author's hand-written declaration of which classes the catalogue dispatches on as top-level credential subjects.
It cannot be extracted from the ontology, because trust-framework ontologies do not mark classes as "catalogue roles":

* In Gaia-X 2511 specifically (`gx-2511`, the only framework currently shipped), every `gx:` subject in the ontology is exactly one of `owl:Class`, `owl:DatatypeProperty`, or `owl:ObjectProperty`.
There is no `gx:CatalogueRole` superclass, no `isRole` annotation property, no skos scheme that groups Participant + ServiceOffering + Resource together.
The 213 named OWL classes are structurally indistinguishable from one another as far as the model is concerned — Address, Endpoint, RegistrationNumber and Participant are peers in the type hierarchy.
* `skos:inScheme` exists on most classes but is purely topical categorisation (legal, storage, compute, …); the `gx:participant` scheme happens to contain only Participant, but `gx:legal` contains 37 classes that are not roles.
* JSON-Schema-based frameworks (e.g. UNTP, see ADR 12) have no OWL ontology at all; the role list is declared explicitly under `types:` for the same reason.

Where role-ness *is* encoded — outside the ontology — is in three operative places: the conformity body's issuer behaviour (which `credentialSubject.type` values it actually mints as top-level VCs), the framework's architecture/compliance specification prose, and sample VPs published by the conformity body.
None of these are machine-readable from the catalogue's perspective.

The practical consequence: onboarding a new trust framework is a curation task.
The bundle author reads the framework's spec and conformity-body docs, identifies the small set (typically three to ten) of credential types the catalogue should surface, and writes them into `roles:`.
The list is short by design and stable across ontology revisions; subclass-closure indexing handles the rest of the type hierarchy automatically.

**Role indexing (SHACL bundles):** For each declared role, the registry takes the root URI as `namespace + roleName`, then walks DOWN the OWL hierarchy in `ontology.ttl` via `rdfs:subClassOf*` and indexes every reachable class under that role.
The optional `additional_roots` list adds non-subclass URIs (siblings, cross-hierarchy mappings) to the same role.
Bundles whose `validation_type` is not `shacl` are loaded but deferred (no validation engine wired yet); a WARN is logged.

The Loire 2511 bundle declares `gx:DigitalServiceOffering` as an `additional_root` of the `ServiceOffering` role.
The decision is defensive: the upstream specification is internally inconsistent on DSO — the ontology declares the class abstract (`skos:definition` "the virtual base class for any kind of digital service (for inheritance not for instanciation)"), defines no `rdfs:subClassOf gx:ServiceOffering` link and no concrete subclasses, yet the SHACL layer assigns it its own `sh:targetClass`.
A `rdfs:subClassOf*` walk from `gx:ServiceOffering` therefore never reaches DSO, and routing has to be added explicitly if the catalogue is to accept DSO credentials at all.
Empirically the conformity body does not currently issue DSO at the top level (the Loire compliance dispatcher has no DSO-specific validation component, the published compliance criteria do not cover DSO, and the official Loire test presentations contain only `gx:LegalPerson`, `gx:Issuer`, and `gx:VatID` under the Participant family).
The mapping is therefore inactive in practice today and exists as a zero-cost safety net so DSO credentials, if they ever appear, land somewhere rather than failing with "no proper CredentialSubject found."

**`TrustFrameworkRegistry` (new bean, registered by `TrustFrameworkRegistryConfig`):**

* `resolveRole(typeUri): ResolvedRole` — returns `(frameworkProfileId, role)` for a credential subject's type URI, or `ResolvedRole.UNKNOWN` if no bundle claims the type
* `getActiveBundles()` — collection of bundles whose validation engine is wired (deferred bundles excluded)
* `getBundle(profileId)` — bundle lookup
* `getEffectiveRoles(profileId)` — role set declared by a bundle
* `isFrameworkEnabled(profileId)` — true for SHACL-active bundles

**`TrustFrameworkService` (new bean):** mediates the DB-backed enabled flag per framework family (table `trust_frameworks`).
Other bounded contexts inject this service instead of the repository directly — preserving the bounded-context boundary.
`LoirePolicyEnforcer` consults `isEnabled("gaia-x")` before applying Loire-specific policies; the admin API uses `setEnabled(family, enabled)` to toggle.

**`CredentialFormatProcessor` chain (new):** replaces the previous inline format detection in `CredentialFormatDetector` and the earlier `FormatMatcher` interface.
Each processor is a `@Component @Order(n)` bean owning the full lifecycle for one credential family — `match` (classification), `process` (outer-envelope unwrap, JWT signature verification when applicable, format-specific policy enforcement), and `unwrapNested` (nested-payload unwrap for inner VCs).
`CredentialFormatDetector` is now an empty orchestrator that iterates the chain and returns the first non-`UNKNOWN` classification.
The current implementations are `LoireCredentialProcessor` (Gaia-X ICAM 24.07) and `Vc2DanubeTechCredentialProcessor` (W3C VC 2.0 JSON-LD / JWT); adding a new credential format requires only a new processor implementation, no edits to the detector or the verification strategy.

**`LoirePolicyEnforcer` (new bean):** owns Loire-specific verification policies (issuer `did:web` restriction, `x5u` trust-chain validation against the Gaia-X Trust Anchor Registry).
Reads the Trust Anchor Registry URL from its bundle's `properties.trust_anchor_url` — the configuration travels with the bundle rather than living in `application.yml`.
The enforcer is a no-op when the `gaia-x` family is disabled, so any processor can call `enforceIfApplicable()` without pre-checking enablement.

**`TrustFrameworkBaseClass` enum:** removed.
Call sites use `ResolvedRole` (a record carrying `frameworkProfileId: String` and `role: String`).
Storage and API routing dispatch on the role string directly; the bundle config — not Java source — is the authority for what roles exist.

**`CredentialFormat`:** the existing `GAIAX_V2_LOIRE` / `VC2_DANUBETECH` enum continues to drive format detection (JWT structure, header validation, unwrapping).
It is independent of role resolution: any credential whose subject type URI is registered by some bundle resolves to that bundle's role, regardless of which JWT format envelope carries it.

**Multi-framework side-by-side:** Multiple bundles load simultaneously.
Namespaces are disjoint by construction; resolution is unambiguous.
A VP containing credentials from different frameworks routes each credential through its own bundle's role.
Adding a new framework is a bundle drop-in — either as a classpath resource baked into a custom JAR or as a directory under the deployment-time overlay path, in both cases containing `framework.yaml` plus the ontology/SHACL files; no Java changes.

**Removed configuration:** the `federated-catalogue.verification.{participant,resource,service-offering}.type` properties (and the `service-offering.additional-types` workaround) are no longer read.
Type URIs are now declared exclusively in each bundle's `framework.yaml`.

**Result materialization:**

The verification core returns a single generic result schema across all roles; the `role` field on the result carries the bundle-declared role name (e.g. `Participant`) and consumers dispatch on that string.
There is no per-role Java result type and no `result_type` declaration in `framework.yaml`: the role name itself is the dispatch key, and adding a new role to a bundle requires no Java change.

**Out of scope for this ADR:**

* JSON Schema validation engine wiring (bundles with `validation_type: json-schema` load but defer until the engine is integrated)
* A second framework bundle delivered alongside the catalogue — the mechanism exists; only the `gaia-x-2511` bundle ships at present
* A generic type-URI-based query API — the existing role-based endpoints (`/participants`, `/service-offerings`, `/resources`) cover current needs; future frameworks that define different role names can expose them via framework-specific endpoints or a generic type query

**Rationale:**

* Fully bundled framework definitions make adding `gx:DigitalServiceOffering` (or any future sibling) a one-line change in `framework.yaml` — no Java, no redeploy logic.
* Removing the enum eliminates the architectural coupling between Gaia-X ontology structure and Java source; the verification pipeline is now a generic engine driven by bundle metadata.
* Bundle-per-framework gives a clear extension point — a developer adding a new framework drops a directory; the registry treats it identically to the bundled one.
* Explicit role declarations (rather than `sh:targetClass` auto-discovery) keep the bundle's public surface minimal and reviewable: a reader sees three role names, not the full transitive subclass closure.
* Supersedes the hardcoded `loireBaseClassUris` / `legacyBaseClassUris` maps and `service-offering.type` / `participant.type` / `resource.type` properties introduced in ADR 9.
* The typed result POJOs `CredentialVerificationResultParticipant`, `CredentialVerificationResultOffering`, and `CredentialVerificationResultResource` were removed without an adapter or deprecation window.
No known external Java consumers of these typed return values exist: the catalogue's public interface is its REST/OpenAPI endpoints and NATS events, not Java class types.
An adapter phase would delay removal without providing safety to any known consumer.

=== ADR 15: Metadata Enrichment of Non-RDF Assets

**Context:** Non-RDF assets (PDFs, images, plain JSON, …) are stored as opaque blobs.
Their content is indexed only in the File Store and metadata columns; they are invisible to SPARQL/openCypher queries.
Operators need a way to annotate non-RDF assets with machine-readable, graph-queryable metadata (e.g., type, classification, provenance triples) without replacing the underlying file.

Two approaches were considered:

1. **Separate enrichment endpoint** (`POST /assets/{id}/metadata`): explicit, avoids routing ambiguity.
Adds a new endpoint and a new service path.
2. **Unified `POST /assets` with subject-IRI routing:** reuse the existing upload endpoint; when the uploaded RDF's primary `@id` matches an existing non-RDF asset, route to the enrichment path.
No new endpoint required.

**Decision:** Option 2 — subject-IRI routing at `POST /assets`.

* When uploaded RDF is detected and the asset store reports an existing non-RDF asset for the payload's primary subject IRI, the request is routed to the enrichment path.
The lookup also enforces that `HUMAN_READABLE` assets cannot be enriched — they are rejected with HTTP 422.
* The enrichment path validates that the payload contains at least one triple whose subject is the asset's IRI, filters protected-namespace triples (`ProtectedNamespaceFilter`), replaces the asset's graph triples (`deleteClaims` → `addClaims`), and persists the raw RDF to `assets.content`.
The change is captured in the Hibernate Envers audit trail with `changeComment = "Metadata enriched"`.
Returns HTTP 200.
* Each enrichment upload fully replaces the previous metadata (overwrite, no merge).
This matches the existing RDF version-update semantics.
* The `content_kind` column is never changed by enrichment.
It always records the original upload modality.
* Storing the enrichment RDF in `assets.content` means the existing PostgreSQL→graph rebuild process picks up enriched non-RDF assets automatically — no extra rebuild logic needed.
* Enrichment requires an active graph backend.
If `graphstore.impl=none`, the request fails with HTTP 503 before any state change — graph writes are mandatory for this path.

**Rationale:** Reusing the existing endpoint avoids a new API surface and keeps the upload contract consistent.
Routing on `content_kind` is unambiguous and preserves backward compatibility — uploads for RDF assets and non-RDF assets are unaffected.
Overwrite semantics simplify the data model: no partial-merge conflicts, no ordering dependencies between enrichment uploads.

**Related:** ADR 4, ADR 7.

=== ADR 16: Schema-module admin toggles, with OWL gating role resolution

**Related:** ADR 10.

**Context:** Four schema languages — SHACL, JSON Schema, XML Schema, OWL — must each be enable/disable-able at runtime through the administration UI, and every toggle must have a demonstrable effect.

SHACL, JSON Schema, and XML Schema all fit a uniform validator shape: input runs against stored schemas of the matching type and a conformance verdict is returned.
OWL is different — uploaded ontologies extend the type vocabulary the role resolver consults via `rdfs:subClassOf+`; OWL has no validation step of its own.
A uniform `module_disabled` semantic therefore does not fit OWL: without separate semantics, the OWL toggle would persist in the database with no observable effect, which violates the demonstrable-effect requirement.

**Decision:** Expose all four toggles on the admin schema-modules page, but treat them as two semantic classes. The admin UI splits the two classes into separate sections so the asymmetry is visible to operators.

* **Validators (SHACL, JSON Schema, XML Schema):** the toggle gates the validator wherever the catalogue would otherwise run it — on-demand validation, and credential verification when the caller requests a schema check.
When off, the affected endpoint returns `400 Bad Request` with body `module_disabled:<MODULE>`.
Schema storage is unaffected; schemas remain queryable through the schema endpoints regardless of toggle state.

* **OWL:** the toggle gates the third tier of role resolution — the `rdfs:subClassOf+` walk over runtime-uploaded ontologies, on top of the registry index and `additional_roots` defined by ADR 10.
When off, only tiers 1 and 2 are consulted; credentials whose type is only reachable through the subclass walk resolve to `ResolvedRole.UNKNOWN`, and the verification entry point rejects them with `400 Bad Request`.
Registry-direct types (Participant, ServiceOffering, Resource for the Gaia-X 2511 bundle, and equivalents in any other bundle's `framework.yaml`) are unaffected.

**Consequences:**

* Disabling a validator is *permissive* (a check is skipped); disabling OWL is *restrictive* (the catalogue stops recognising custom subclasses).
The asymmetry is intentional: gating role resolution is the strongest demonstrable effect for a type-vocabulary input, and a uniform "skip a check" semantic does not exist for OWL.

* The OWL toggle is independent of the caller's `verifySchema` and `verifySemantics` flags.
Type dispatch is a configuration of the catalogue's type system, not a per-request validation step, so disabling OWL affects every internal caller of the verification path — asset upload, provenance ingest, asset metadata flow, base subscriber, and on-demand role resolution — independent of how those callers parameterise their request.

* The SHACL toggle also gates the periodic revalidation sweep over the existing claims graph.
When SHACL is administratively disabled, `RevalidationServiceImpl.handleTask` short-circuits — the sweep skips per-asset validation rather than revoke previously-conforming assets against shapes the operator has explicitly silenced.
The periodic sweep is not actively wired in the default production configuration today; the gate is pre-positioned for when it is.
Re-sweeping the backlog of chunks skipped while SHACL was off (re-queueing them on the false→true transition) is a follow-up that lands together with production wiring of the sweep — it is a one-liner against `RevalidationService.startValidating()` at that point.

* On-demand validation (`POST /assets/validate`) rejects with `400 Bad Request` not only for `module_disabled:<MODULE>` but also for the broader "no validation module is enabled or applicable" case.
Both are administrative-configuration mismatches under operator control, not server-side processing failures, so `ClientException` (400) is the appropriate mapping rather than `VerificationException` (422). 422 remains reserved for assets whose type is not validatable at all (not RDF, unsupported content type).
