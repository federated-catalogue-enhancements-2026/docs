== Cross-cutting Concepts

=== Backup
Backing up a running application is a basic administrative operation. The recommended runtime environment for the federated catalogue is Kubernetes. See the previous section for more information. Therefore, to back up the federated catalogue, well known tools can be used.

Data is persisted in the following components:

* Federated Catalogue Server: Asset and Schema files (Filesystem)
* Metadata Store: Asset and Schema metadata (PostgreSQL)
* Graph store: Claims graph (Neo4j or Fuseki, depending on the configured backend)

During the deployment it must be ensured that volumes are mounted to store the persistent data in the mentioned components.

The backup can be configured for those volumes as a whole, on a regular basis. In case of any problem, the system can be stopped and completely restored the volume  from its latest backup version.

The details for configuring the backup as well as the backup encryption depends on the Kubernetes distribution, where the service is deployed (e.g., for OpenStack based Kubernetes distributionsfootnote:[https://docs.elastx.cloud/docs/openstack-iaas/guides/volume_backup/[Volume Backup & Restore | Documentation]]).

=== Logging
The components use ```Slf4j``` for logging. ```Logback``` will be used as implementation.

As default, all components should log to Standard out. Log aggregation is realized by the surrounding environment

=== Trust Framework Bundles

Each supported trust framework is delivered as a self-contained bundle (`framework.yaml` + ontology + SHACL shapes), either packaged with the JAR or supplied at deployment time as a filesystem directory, under `trustframeworks/<bundle-id>/`.
A `TrustFrameworkRegistry` bean indexes all bundles at boot and resolves credential subject types to bundle-defined base classes, replacing the previous hardcoded URI-to-enum mapping.
See ADR 10 for the design rationale and <<_trust_framework_bundles_and_base_class_resolution,Building Block View — Trust Framework Bundles and Base-class Resolution>> for the runtime structure.

==== Onboarding a Trust Framework: Identifying Base Classes

The bundle's `base_classes:` map declares the small curated set of root classes that the catalogue dispatches on as top-level credential subjects.
Subclass-closure indexing (`rdfs:subClassOf*`) handles the rest of the type hierarchy automatically; the curation effort is limited to picking the roots.

A non-obvious property of this design: **the base-class list cannot be derived from the framework's ontology**.
It is a manual, judgment-based curation step that every new bundle author has to perform, and the source-of-truth lies *outside* the ontology file.

*Why the ontology cannot answer the question.* Trust-framework ontologies model the vocabulary — every class that can appear anywhere in a credential's claims graph (legal entity attributes, addresses, endpoints, value objects).
They do not model which classes are issued as standalone top-level credentials.
Ontologies do not mark base classes: there is no `:isBaseClass` annotation, no `:CatalogueBaseClass` superclass, no SKOS scheme that groups base classes together.
Topical schemes (legal, storage, compute, …) categorise classes by domain and contain many non-base-class members.
The base-class-bearing classes are structurally indistinguishable from value-object classes in the model.
Three mechanical proxies — "every named OWL class", "every `sh:targetClass` in the SHACL shapes", "every class whose only superclass is `owl:Thing`" — all over- or under-include in practice.
JSON-Schema-based frameworks have no OWL ontology at all; the base-class list is declared explicitly under the bundle's `types:` field for the same reason.

*Where base-class-ness is encoded.* Outside the ontology, in three operative places — none of them machine-readable from the catalogue's perspective:

* The conformity body's issuer behaviour — which `credentialSubject.type` values it actually mints as top-level VCs.
Discoverable by inspecting the conformity body's compliance components and the credentials it returns.
* The framework's compliance and architecture specifications — the per-credential-family compliance criteria.
The presence of a "criteria for X" document is strong evidence that X is a top-level base class; the absence of one for a class — even one that has a SHACL shape — is strong evidence the conformity body does not issue it.
* Sample verifiable presentations published by the conformity body.
Decoding the inner VCs of the official test VPs reveals the credential types that actually appear in production traffic.

*Onboarding recipe.* Read the framework's spec to find the section listing top-level credential types.
Cross-check against the conformity body's component architecture (one validation service per top-level credential family is the typical pattern) and against the conformity body's published sample VPs.
Where the three sources agree, you have the base-class list.
Where they disagree, prefer what the conformity body operatively does over what the spec aspirationally describes.
The resulting list is short by design (typically three to ten entries) and stable across ontology revisions.

*Spec inconsistencies are normal and expected.* Two patterns recur:

* A class whose `skos:definition` declares it abstract ("for inheritance not for instantiation") but which carries its own `sh:targetClass` shape — the shape implies a valid credential subject; the prose implies the opposite.
* A class that is conceptually a specialisation of another but the ontology does not declare an `rdfs:subClassOf` link between them, often because they live in different parts of the namespace or were modelled independently.

When the conformity body is silent on such a class — no validation component, no criteria document, no test payload — the safe default is a defensive mapping: route the class to the most semantically adjacent existing base class via `additional_roots` so credentials of that type, if they ever appear, land somewhere rather than being rejected.
Document the mapping inline with a pointer to what would un-block dropping it (typically: confirmation from the conformity body that the class is never issued at the top level).

==== Deploying Bundle Overlays

A bundle overlay lets operators customise an existing bundle or add a new one at deployment time, without rebuilding the JAR.
The operator points the catalogue at a read-only directory mounted into the container; its layout mirrors the classpath bundle layout (one sub-directory per bundle, containing the framework descriptor and optional ontology and shapes files).

When the bundle identifier in a filesystem descriptor matches a classpath bundle, the file is treated as an overlay and only needs to declare the identifier plus the fields to change.
Nested maps merge recursively (filesystem wins per leaf key); lists and scalars replace wholesale when the overlay supplies them; sibling ontology and shapes files are inherited from the classpath bundle when the overlay directory omits them.
A filesystem entry whose identifier has no classpath match is loaded as a brand-new bundle and must declare all mandatory fields.
See ADR 10 for the full deep-merge rules and the operator guide for the exact configuration surface.

==== Runtime Override Precedence

The filesystem overlay above is a deployment-time mechanism; it requires editing a mounted file and restarting (or reloading) the catalogue.
For the subset of bundle properties that are *external client identifiers* (client implementation key, compliance service endpoint, per-request timeout, trust-anchor list location, etc.) a second, runtime-only override path exists — see ADR 19.

Resolution order at every compliance call is **DB override > bundle YAML > hard-coded default**, performed by `TrustFrameworkProfileResolver` and returned as the effective configuration to the compliance pipeline.
The registry continues to serve role resolution and bundle discovery; it does not consult the DB.

The DB-override layer is for administrators changing endpoint identifiers without redeployment.
The filesystem overlay remains the right tool for bundle-shape changes (roles, ontology, validation type) and for shipping a new bundle.

==== Type Dispatch and the OWL Toggle

When a credential arrives, the catalogue maps its `credentialSubject` type URI to a trust-framework base class through three tiers, in order:

1. **`TrustFrameworkRegistry.typeIndex`** — a hash lookup populated at boot by walking each bundle's `ontology.ttl` from every declared base class down via `rdfs:subClassOf*`. Hits here cover every class statically reachable from a base class at startup.
2. **`additional_roots`** — sibling URIs declared in `framework.yaml` for cases where the ontology's class graph does not connect the sibling to the base class (see ADR 10 for the `gx:DigitalServiceOffering` example). Also indexed at boot.
3. **Composite-ontology subclass walk** — `ClaimValidator.resolveSubjectBaseClass` runs a SPARQL `rdfs:subClassOf+` query against the runtime composite ontology assembled from `ONTOLOGY`-type schema rows. This is the only tier that consults schemas uploaded *after* boot.

The OWL administrative toggle (ADR 16) controls whether the catalogue admits **runtime extensions** to its registered type vocabulary.
With OWL on, custom subclasses introduced through runtime-uploaded ontologies are recognised and routed to the registered base class they extend.
With OWL off, the catalogue accepts only the types statically known at boot — credentials whose type was introduced via a runtime ontology upload are rejected at the verification entry point with `400 Bad Request`.
Tiers 1 and 2 continue to run in either state; only tier 3 (the subclass walk over the runtime composite ontology) is gated.
The admin UI labels the toggle accordingly, so operators see the user-facing effect rather than the implementation tier it gates.

Disabling OWL therefore has a *narrow* but *strict* effect: framework-direct types (Participant, ServiceOffering, Resource for the Gaia-X 2511 bundle, and equivalents in any other bundle's `framework.yaml`) keep resolving and keep being accepted; custom subclasses defined exclusively in a runtime-uploaded ontology stop resolving and are rejected.
Gating base-class resolution is the most visible effect a toggle on a type-vocabulary input can have, and the one that satisfies the demonstrable-effect requirement for runtime toggles.

==== Gaia-X Loire v2 Compliance Service

The shipped Gaia-X compliance bundle wires a generic JWT-VC compliance client to the canonical GXDCH front-end.
The client implementation is wire-shape-specific (it POSTs a Verifiable Presentation JWT and expects a Verifiable Credential JWT in response) but path-agnostic; the endpoint URL and request path are declared per bundle.
This keeps the Gaia-X Loire / GXDCH endpoint as the reference deployment while allowing any future framework that adopts the same exchange to onboard as a new bundle, with no new Java code required.
See ADR 12 for the wire-shape-in-bundle rationale.

The catalogue talks to the canonical Gaia-X compliance endpoint, which is a single URL maintained by the Gaia-X Digital Clearing House (GXDCH).
How that endpoint distributes traffic across certified GXDCH operators is a concern of the GXDCH infrastructure itself (see the `loadbalancer/` configuration in the upstream https://gitlab.com/gaia-x/lab/gxdch[GXDCH repository]) and is intentionally out of scope here; from the catalogue's perspective there is one URL, carried in the bundle descriptor, and no per-operator configuration.

**Request / response shape.** The catalogue POSTs a Loire-conformant Verifiable Presentation JWT to the configured compliance path appended to the service URL, carrying the VP JWT's `id` claim as a query parameter.
The endpoint validates the VP against Loire 2511 criteria and returns a Compliance VC JWT signed by the selected operator.
No authentication header is required — the GXDCH endpoint is publicly accessible.
The OpenAPI specification of the compliance service is authoritative for the exact request and response shape.

The returned Compliance VC JWT is persisted as-is alongside the asset.
See ADR 17 for the storage rationale.

**Test coverage.** Live GXDCH coverage requires a deployment stage whose signing certificate is reachable via a public `x5u` URL (the Trust Anchor Registry must be able to resolve the chain).
The integration-test suite includes BDD scenarios that exercise the live path; they are excluded from default CI and run only against a QA stage with a resolvable certificate chain.

=== Admin Configuration Persistence

Admin configuration is split across four kinds of persisted state:

* **Family-level enable/disable.** One row per trust-framework family known at deployment time; the row carries the operator-toggleable enabled flag and identifies the family.
* **Per-bundle base-class state.** One entry per (bundle, base-class) pair the operator has explicitly disabled; absence of an entry means the base class is enabled by default.
* **Per-bundle runtime overrides** of the external client identifiers declared in the bundle YAML.
Each entry is keyed by the bundle profile ID; a missing override for a given identifier means "use the YAML value".
See ADR 19 for the resolver and precedence model.
* **Generic key-value runtime settings** (schema-module toggles, preferred graph backend, and similar) that do not warrant their own table.

The authoritative column-level definitions live in the Liquibase changesets; this section is a conceptual map of which kind of state lives where.

=== Admin API Mutation Convention

Admin resource mutations follow a single convention: `PATCH` of the resource with an `application/merge-patch+json` body containing only the fields being modified.
Empty patches are rejected at the API boundary; absent fields are left unchanged.

Two flavours of merge-patch coexist:

* **Strict typed patch** — used for resources whose only mutable state is a boolean toggle (family enabled flag, base-class enabled flag, schema-module enabled flag).
The body is a fully-typed DTO; unknown keys are rejected.
JSON `null` carries no useful meaning for a boolean, so absent and explicit-null are not distinguished.

* **RFC 7396 patch with explicit-null clearing** — used where the resource carries operator-set overrides that need a non-destructive revert path (currently per-bundle runtime overrides; see ADR 19).
Setting a recognised key to a value persists it as an override; setting it to JSON `null` clears that override and lets the baseline flow through again; omitting a key leaves the existing state untouched.

A wholesale revert via `DELETE` on the bundle override resource exists for the "back to baseline for this whole bundle" case; it is idempotent.

The OpenAPI spec is authoritative for which endpoints follow which flavour.
The earlier `PUT /…/enabled?enabled=<bool>` shape used during initial admin-API development is no longer accepted.

=== Input Parsing Security

User-supplied content (assets, schemas, IRIs) is parsed only by hardened, sandboxed parsers. The catalogue treats every external byte sequence as untrusted, regardless of the caller's role. The same posture is applied uniformly across validators and graph stores.

[options="header",cols="1,2,2"]
|===
| Threat | Mitigation | Component
| XXE / external DTD | JAXP secure-processing enabled; external-DTD and external-schema access disabled. | XML schema validator, XML parsers
| SSRF via JSON `$ref` | `$ref` URI scheme allowlist: only `https://` is loaded; `file://`, `http://`, `ftp://`, and similar are rejected before schema load. | JSON schema validator
| SPARQL injection | IRIs validated against a strict syntax check before string-interpolation into SPARQL queries. | SPARQL graph store
|===

Any new validator or store that accepts user-provided content must adopt the same posture. Hardening is enforced by tests: every new parser has at least one security-path test (malicious DTD, external `$ref`, unsafe IRI) alongside its happy-path tests.

=== Asset Content Kinds and Metadata Enrichment

Two independent classifications apply to every asset:

* **`content_kind`** (`RDF`, `NON_RDF`) — the **format** of the originally uploaded payload.
* **`AssetType`** (`MACHINE_READABLE`, `HUMAN_READABLE`, or `NULL`) — the **role** in an MR-HR pair, with a content-format constraint on the MR side. `MACHINE_READABLE` means the payload is itself machine-parseable: RDF (JSON-LD credential, shapes graph, …) or a structured non-RDF format (JSON, CSV, XML).
Opaque binaries (PDFs, images) are never `MACHINE_READABLE`. `HUMAN_READABLE` is the human-facing counterpart (PDF, HTML, DOCX, plain text). `NULL` means the asset is unpaired.

These axes are orthogonal, so all five combinations below are valid states; the table records what each means and whether the asset is a candidate for metadata enrichment.

[options="header",cols="1,1,2"]
|===
| content_kind | AssetType | Meaning
| `RDF` | _null_ | Standalone RDF asset (credential, ontology, shapes graph, …)
| `RDF` | `MACHINE_READABLE` | RDF asset acting as the structured counterpart of an MR-HR pair
| `NON_RDF` | _null_ | Standalone non-RDF binary/text asset (PDF, JSON, image, …); may be enriched
| `NON_RDF` | `MACHINE_READABLE` | Non-RDF structured counterpart of an MR-HR pair (e.g., CSV paired with a PDF); may be enriched
| `NON_RDF` | `HUMAN_READABLE` | Human-facing attachment of an MR-HR pair; **cannot be enriched**
|===

NOTE: Enrichment eligibility is decided by `content_kind = NON_RDF` and `AssetType ≠ HUMAN_READABLE`.
The `MACHINE_READABLE` constraint means the payload is parseable (structured non-RDF or RDF); enrichment specifically attaches *graph-queryable* RDF claims on top of a payload that was not RDF at upload time, regardless of whether the payload was otherwise structured.

**Metadata enrichment** is the process of attaching graph-queryable RDF triples to a non-RDF asset.
An RDF payload is uploaded against the IRI of an existing non-RDF asset.
The service:

. Parses the payload and requires at least one triple whose subject is the asset's IRI.
. Filters out triples in protected namespaces.
. Replaces the asset's existing graph triples.
. Persists the raw RDF payload alongside the existing non-RDF asset (no new entity is created and no separate enrichment table is involved).
The audit trail records each enrichment as a new revision, so the version history of the asset includes every enrichment, and the graph can be rebuilt from the relational store at any time.

`HUMAN_READABLE` assets reject enrichment.
When the graph backend is disabled, enrichment is rejected as well — graph writes are mandatory for this path.
The asset's content kind is not changed by enrichment, and each enrichment upload replaces the previous metadata entirely (overwrite semantics — no merge).
The OpenAPI specification is authoritative for the exact HTTP status codes returned in each case.
